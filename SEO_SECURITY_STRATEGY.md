## PART 1: GLOBAL SEO STRATEGY SUMMARY

### Brand Positioning
**Vynex Media** is a premium, boutique visual production agency in Amman, Jordan, specializing in high-quality product photography, restaurant/café shoots, and short-form video content creation for social media platforms. We position ourselves as the bridge between local brands and scroll-stopping digital content.

**Tagline:** "نخلي محتواك يصوّر قصتك" (We make your content tell your story) + "Visual Media Agency Turning Your Ideas Into Scroll-Stopping Content"

---

### Target Audience Primary Personas

1. **Restaurant & Café Owners** (عرب 50%)
   - Ages 28–50, Arabic-speaking
   - Need: Professional food photography, Instagram Reels, menu promotion
   - Pain Point: Lack of professional content for social media

2. **E-commerce & Product Brands** (عرب & International 30%)
   - Online store owners, drop-shippers, boutique brands
   - Need: Product photography, lifestyle shots, short-form video demos
   - Pain Point: Cost of professional photography, tight turnaround

3. **Marketing Agencies & Corporate** (عرب 15%)
   - Marketing agencies, corporate social media teams
   - Need: Bulk content production, consistent quality, quick turnaround
   - Pain Point: Finding reliable, professional local production partners

4. **Influencers & Content Creators** (عرب 5%)
   - Micro and macro influencers, YouTube creators
   - Need: Professional shoot facilitation, editing, post-production

---

### Primary Business Goals (Ranked)
1. **Lead Generation** → Contact form submissions, WhatsApp inquiries
2. **Shoot Bookings** → Direct product, food, and video shoot reservations
3. **Brand Authority** → Establish Vynex as *the* go-to agency in Amman
4. **Repeat Business** → Build long-term client relationships

```
│   ├── /blog/post-slug
├── /contact (تواصل-معنا)
├── /faq (الأسئلة-الشائعة)
└── /testimonials (شهادات-العملاء)
```

### URL Strategy
- **Format:** Lowercase, kebab-case, descriptive, concise (max 50 chars)
- **Arabic:** Use transliteration for URLs OR English versions with Arabic content
- **Recommendation:** Use English URL slugs with Arabic page titles and content for better SEO
- **Trailing slash:** Use trailing slashes for consistency (`/services/` not `/services`)

## PART 8: SECURITY HARDENING PLAN

### LAYER 1: INFRASTRUCTURE & HOSTING

**1.1 HTTPS & TLS**
- [ ] Obtain SSL/TLS certificate (Let's Encrypt free or premium CA)
- [ ] Minimum TLS 1.2 (prefer TLS 1.3)
- [ ] Certificate validation: Domain validation (DV) at minimum
- [ ] Wildcard cert recommended: `*.vynexmedia.com`
- [ ] Auto-renewal: Set up with ACME (Let's Encrypt)
- [ ] Redirect all HTTP → HTTPS (301 permanent)
- [ ] HSTS Header: `max-age=31536000; includeSubDomains; preload`

**1.2 Web Application Firewall (WAF)**
- [ ] Consider Cloudflare Free Plan (DDoS protection, WAF rules)
- [ ] Alternative: AWS WAF if hosting on AWS
- [ ] Rules to implement:
  - [ ] Rate limiting: 10 requests/sec per IP
  - [ ] SQL injection protection: Block common payloads
  - [ ] XSS protection: Sanitize input patterns
  - [ ] Geo-blocking: Optional (allow only Middle East if desired)
  - [ ] Bot protection: Challenge suspicious traffic
- [ ] Monitor WAF logs daily (first month)

**1.3 DDoS Protection**
- [ ] Use CDN with DDoS protection (Cloudflare, AWS CloudFront)
- [ ] Rate limiting: Drop IPs with >100 requests/min
- [ ] CAPTCHA on suspicious traffic patterns
- [ ] Monitor traffic spikes (alert if 5x normal)

**1.4 Server Hardening (If Self-Hosted)**
- [ ] OS patches: Apply security updates within 48 hours
- [ ] SSH hardening:
  - [ ] Disable password auth (SSH keys only)
  - [ ] Change default port 22 → random port 20000-30000
  - [ ] Disable root login
  - [ ] Use `fail2ban` for brute-force protection
- [ ] Firewall: UFW or iptables
  - [ ] Allow only: SSH, HTTP (80), HTTPS (443)
  - [ ] Block all other inbound ports
- [ ] File permissions: Use principle of least privilege
  - [ ] Web root: 755 for dirs, 644 for files
  - [ ] No files with 777 permissions

**1.5 Database Security**
- [ ] Database in private subnet (not accessible from internet)
- [ ] Strong password: min 16 chars, special chars, numbers
- [ ] Separate DB user (not root): Minimal privileges
  - [ ] User can only SELECT, INSERT, UPDATE on specific tables
  - [ ] No CREATE, DROP, ALTER privileges
- [ ] Database encryption at rest (if available)
- [ ] Database encryption in transit (SSL connection)
- [ ] Regular backups (see Section 1.7)
- [ ] No credentials in code (use environment variables)

**1.6 Backups & Disaster Recovery**
- [ ] Backup frequency: Daily (automated)
- [ ] Backup location: Off-site (S3, Google Cloud, different provider)
- [ ] Backup retention: 30-day rolling window (delete after 30 days)
- [ ] Test restore: Monthly full restore to staging
- [ ] Document recovery procedure:
  - [ ] Time-to-restore target: 1-2 hours
  - [ ] Steps to restore from backup
  - [ ] Contact info for backup service
- [ ] Backup encryption: AES-256
- [ ] Versioning: Keep multiple backup versions

**1.7 Logging & Monitoring**
- [ ] Web server logs: Nginx/Apache access logs
  - [ ] Log format: Include IP, timestamp, path, status code, user agent
  - [ ] Retention: 30 days
  - [ ] Monitor for 4xx, 5xx spikes
- [ ] Application logs: Error logs, warning logs
  - [ ] Log to separate file
  - [ ] Alert on ERROR or CRITICAL
- [ ] Security logs: Failed login attempts, suspicious patterns
  - [ ] Monitor login failures (alert if 10+ failed in 1 hour)
  - [ ] Monitor API abuse (same endpoint >100/min)
- [ ] Intrusion detection: Fail2ban or similar
  - [ ] Alert on repeated 404s, SQL injection patterns
  - [ ] Auto-ban IPs with >50 failed requests/hour
- [ ] Email alerts: Critical errors → admin@vynexmedia.com (within 15 min)
- [ ] Log aggregation (optional): ELK Stack, Splunk, or Datadog

### LAYER 2: APPLICATION SECURITY (OWASP Top 10)

**A1: Injection (SQL, NoSQL, Command Injection)**
- [ ] Use parameterized queries / ORM (e.g., Sequelize, Prisma for Node)
- [ ] Never concatenate user input into SQL
- [ ] Validate & sanitize all inputs (whitelist approach)
- [ ] Use prepared statements with placeholders
- [ ] Example (bad): `SELECT * FROM users WHERE id = ${userId}`
- [ ] Example (good): `SELECT * FROM users WHERE id = ?` (parameterized)

**A2: Broken Authentication**
- [ ] Password requirements:
  - [ ] Minimum 8 characters (recommend 12+)
  - [ ] Require uppercase, lowercase, numbers, special chars
  - [ ] Reject common passwords (use password strength library)
- [ ] Session management:
  - [ ] Session tokens: Random, unpredictable (use crypto library)
  - [ ] Session timeout: 30 minutes of inactivity (admin: 15 min)
  - [ ] Secure cookie flags: HttpOnly, Secure, SameSite=Strict
- [ ] Multi-factor authentication (MFA):
  - [ ] For admin panel: Require MFA (TOTP/Google Authenticator)
  - [ ] For user accounts: Optional MFA (email/SMS)
- [ ] Password reset:
  - [ ] Use secure token (not predictable)
  - [ ] Token expires after 1 hour
  - [ ] Send reset link via email only (not SMS)

**A3: Cross-Site Scripting (XSS)**
- [ ] Output encoding: Encode all user-controlled output
  - [ ] HTML: &, <, >, ", ' → HTML entities
  - [ ] URL: Use proper URL encoding
  - [ ] JavaScript: Escape special chars
- [ ] Content Security Policy (CSP):
  ```
  Content-Security-Policy: 
    default-src 'self'; 
    script-src 'self' https://cdn.example.com; 
    style-src 'self' 'unsafe-inline'; 
    img-src 'self' https:;
  ```
- [ ] Sanitization: Input validation library (e.g., DOMPurify for content)
- [ ] No `innerHTML` without sanitization
- [ ] Escape user input in templates

**A4: Broken Access Control**
- [ ] Authorization checks on every endpoint
- [ ] Verify user owns resource before allowing access
- [ ] No "security through obscurity" (guessing IDs)
- [ ] Admin panel: Role-based access control (RBAC)
  - [ ] Admin role: Full access
  - [ ] Editor role: Can edit content, not settings
  - [ ] Viewer role: Read-only
- [ ] Implement middleware to check permissions
- [ ] Example: `if (user.id !== resource.userId) return 403 Forbidden`

**A5: Security Misconfiguration**
- [ ] No default credentials anywhere
- [ ] No debug mode in production
- [ ] No sensitive data in error messages
- [ ] Remove unnecessary services/ports
- [ ] Security headers configured (see next section)
- [ ] CORS properly configured (whitelist origins)

**A6: Sensitive Data Exposure**
- [ ] HTTPS everywhere (covered in Section 1.1)
- [ ] No sensitive data in logs (no passwords, API keys, tokens)
- [ ] Encrypt sensitive data at rest:
  - [ ] Passwords: bcrypt with salt (min 10 rounds)
  - [ ] API keys: Encrypt with master key
  - [ ] Customer data: AES-256 encryption (if storing)
- [ ] No sensitive data in URLs (`?password=123` ❌)
- [ ] Client-side storage: Use secure localStorage (no passwords)

**A7: XML External Entity (XXE)**
- [ ] Disable XML external entities processing
- [ ] If XML upload needed: Use XML parser with XXE disabled
- [ ] Validate against schema

**A8: Broken Object-Level Access Control**
- [ ] Covered above (A4)
- [ ] Verify user can access requested resource

**A9: Using Components with Known Vulnerabilities**
- [ ] Dependency scanning:
  - [ ] Use npm audit / yarn audit (run before deploy)
  - [ ] Tools: Snyk, Dependabot (GitHub)
- [ ] Update dependencies monthly
- [ ] Security patches: Apply within 48 hours
- [ ] Monitor CVE databases
- [ ] Keep framework/CMS updated

**A10: Insufficient Logging & Monitoring**
- [ ] Covered in Section 1.7
- [ ] Log all auth attempts, failed logins, admin actions
- [ ] Monitor and alert on suspicious activity

### LAYER 3: SECURITY HEADERS

Implement all headers in web server config (Nginx, Apache) or middleware:

```nginx
# Security Headers in Nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "geolocation=(), microphone=(), camera=()" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' https://www.googletagmanager.com; img-src 'self' https: data:; style-src 'self' 'unsafe-inline'; font-src 'self' https://fonts.googleapis.com;" always;
```

**Header Explanations:**
- **HSTS:** Force HTTPS (prevents downgrade attacks)
- **X-Content-Type-Options:** Prevent MIME sniffing
- **X-Frame-Options:** Prevent clickjacking
- **X-XSS-Protection:** Browser XSS filter (legacy, CSP is modern)
- **Referrer-Policy:** Control referrer information
- **Permissions-Policy:** Restrict access to device features
- **Content-Security-Policy:** Restrict sources of content
